# Windows portable package experiment

> 状态：本实验已被 **Codex App Manager**（在本地检测能力并管理安装/更新）取代，保留作为技术记录与 manager Windows 便携路径的设计依据，不再作为独立的便携发布产品推进。

## Summary

The extracted Codex Windows payload can be repackaged as a portable ZIP. This is not equivalent to a normal MSIX installation, but it is a plausible fallback for users whose devices block direct MSIX sideload installation while still allowing normal executable launches.

The current prototype is generated by:

```bash
scripts/prepare-windows-portable.sh dist/msix-observe/OpenAI.Codex_26.519.11010.0_x64__2p2nqsd0c76g0.Msix
```

Generated local artifacts:

- `dist/windows-portable/Codex-Windows-x64-portable-26.519.11010.0.zip`
- `dist/windows-portable/SHA256SUMS-windows-portable.txt`

## What the MSIX install provides

The package manifest declares:

- Package identity: `OpenAI.Codex`
- Full-trust desktop entrypoint: `app/Codex.exe`
- `codex://` protocol registration
- Spreadsheet file associations for `.csv`, `.tsv`, `.xls`, `.xlsm`, and `.xlsx`
- Windows visual identity, including display name and logos
- Microsoft Store / MSIX deployment metadata
- Store-managed installation, update, and uninstall behavior

The manifest also disables registry and file-system write virtualization, with explicit exclusions for:

- `$(KnownFolder:LocalAppData)\OpenAI`
- `HKEY_CURRENT_USER\Software\Google\Chrome\NativeMessagingHosts`

That means the packaged app already expects to behave mostly like a normal full-trust desktop app for user data and native messaging writes.

## What the portable package loses

The portable package keeps the application payload layout that Electron expects:

- `Codex.exe`
- `resources/app.asar`
- `resources/codex.exe`
- `resources/node.exe`
- `resources/node_repl.exe`
- bundled browser, chrome, and latex plugins

It loses MSIX registration effects:

- No package identity at runtime
- No Microsoft Store update path
- No `codex://` protocol registration
- No spreadsheet file associations
- No Apps & Features entry
- No Store / winget awareness that Codex is installed
- No MSIX-managed uninstall cleanup

The prototype adds lightweight replacements only for:

- Direct launch: `Run-Codex.cmd`
- Current-user copy into `%LOCALAPPDATA%\Programs\Codex`
- Start Menu shortcut
- Current-user uninstall script

## Computer Use assessment

The base MSIX does not appear to bundle the Computer Use plugin as a normal bundled plugin. The bundled plugin marketplace in this package lists browser, chrome, and latex.

However, the Electron app code does contain Computer Use integration paths and resolves a separately installed plugin root, including:

- `SKY_CUA_SERVICE_PATH`
- `SKY_CUA_NATIVE_PIPE`
- `node_modules/@oai/sky/bin/windows/codex-computer-use.exe`

This suggests Computer Use depends on plugin installation and a native helper executable rather than directly on MSIX package identity.

Expected result:

- If the app launches, can install/enable plugins, and the Computer Use helper executable is allowed to run, Computer Use may work from the portable package.
- If a device policy blocks helper executables, scripts, native pipes, screen capture, or plugin installation, the portable package will still fail.
- This cannot be claimed as equivalent until it is tested on a real Windows machine with Computer Use enabled.

## Recommendation

Do not replace the official MSIX mirror with a custom EXE installer yet.

Recommended publishing order:

1. Keep MSIX as the canonical Windows artifact.
2. Add an experimental portable ZIP artifact for users blocked by MSIX sideload installation.
3. Label the ZIP clearly as "portable / experimental / not Store-managed".
4. Add a Windows smoke-test checklist before promoting it:
   - Launch `Run-Codex.cmd`
   - Sign in
   - Open a local workspace
   - Run a shell command from Codex
   - Install or enable Computer Use
   - Capture screen / control a simple app with Computer Use
   - Test browser and chrome plugins
   - Test Start Menu shortcut from `Install-Current-User.cmd`
   - Test `Uninstall-Current-User.cmd`

An EXE installer can be considered later, but only if it is signed. An unsigned third-party installer wrapper is likely less trustworthy than a ZIP that preserves OpenAI's original signed binaries.
